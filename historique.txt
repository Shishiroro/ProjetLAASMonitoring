Ce qui a ete fait :
- Mode sequentiel ajoute dans LARD : trajectoires d'approche
  temporellement correlees (au lieu de poses independantes)
- Processus Ornstein-Uhlenbeck a dt variable pour :
  alpha_v, alpha_h, yaw, pitch, roll
- Formule : x[i+1] = mean + (x[i] - mean) * exp(-dt/tau) + sigma_d * N(0,1)
- Timeline spatiale : frames uniformes en distance, dt variable
  (l'avion decelere → dt augmente entre les frames)
- Convergence en finale : les deviations OU tendent vers residual (8%)
  pres de la piste. conv(d) = 0.08 + 0.92 * clamp((d-end)/(stab-end), 0, 1)^0.5
- Crab angle : angle de correction vent, converge aussi
- Profils d'approche (ILS, visual, windy, unstabilized) — RETIRE
  dans le nouveau projet, on garde les parametres OU bruts
- Clamp altitude >= ltp_alt + 1.0 (jamais sous la piste)

Phase 2 — Surechantillonnage du processus OU
================================================================
Date : 2026-03-04

Probleme identifie :
- Le processus OU applique frame par frame donnait un aspect
  saccade visuellement (sauts visibles entre frames)

Solution implementee :
- Surechantillonnage interne dans generate_ou_process()
- Nouveau parametre : oversample_factor (defaut 50)
- Entre chaque paire de frames, l'OU simule oversample_factor
  micro-pas avec dt/factor → sauts plus petits, trajectoire lisse
- Les conv_factors sont interpoles lineairement sur les micro-pas
- Sous-echantillonnage final : on ne garde que les frames reelles
  → meme nombre de frames en sortie, mais plus lisses

Fichier modifie :
- [x] project/export/trajectory_builder.py
      - generate_ou_process() : ajout parametre oversample_factor
      - Simulation interne a taux eleve puis sous-echantillonnage

Notes :
- Rendements decroissants au-dela de ~20 (quasi-identique a 50)
- Impact calcul negligeable meme a factor=50
- Aucun impact sur le nombre de frames GES en sortie

Phase 3 — Auto-calcul de tau (correlation_time)
================================================================
Date : 2026-03-04

Probleme identifie :
- correlation_time_s etait un parametre utilisateur dans le XML,
  mais sa valeur realiste depend de l'altitude (modele Dryden).
- A nos altitudes d'approche (~60-300m), tau Dryden vaut ~1-4s.
- L'ancienne contrainte (tau + 5*turb <= 10) etait trop permissive.

Solution implementee :
- Tau retire du XML (plus de parametre utilisateur)
- Contrainte turb_tau_coupling retiree du XML
- Calcul automatique dans Export.py : tau = h / V
  avec h = segment_start_m * tan(3°) (altitude max, glide slope)
  et V = ground_speed_kts * 0.514444
- Formule Dryden simplifiee : a nos altitudes (<1000 ft), L ≈ h
  donc tau ≈ h / V directement

Fichiers modifies :
- [x] project/templates/trajectory_template.xml
      - Retire parametre correlation_time_s
      - Retire contrainte turb_tau_coupling
- [x] project/export/Export.py
      - Ajout calcul auto tau = h_m / speed_ms
- [x] project/export/trajectory_builder.py
      - Valeur par defaut de correlation_time_s mise a 4.0

Valeurs resultantes :
- segment_start=6000m → h=314m → tau=4.2s
- segment_start=3000m → h=157m → tau=2.1s
- segment_start=2000m → h=105m → tau=1.4s

Phase 6 — Refonte parametres OU + couplage roll-lateral (v4)
================================================================
Date : 2026-03-10

Problemes identifies :
- Trajectoires irrealistes : l'avion "zigzaguait" lateralement
  (dev lat max ~83m en turb=1, oscillations rapides)
- Camera secouait trop (pitch ±2°, roll ±10° en turb=1)
- Roll independant de la trajectoire (inclinaison aleatoire)
- Toutes les valeurs OU venaient de LARDv2, calibrees "a l'oeil"

Solutions implementees dans trajectory_builder.py :

1. Tau separes position/attitude
   - tau_position = 5 * tau_attitude (~10s vs ~2s)
   - Position (alpha_h, alpha_v) : derives lentes (inertie avion)
   - Attitude (yaw, pitch, roll) : oscillations rapides (surfaces controle)

2. Std position cales sur specs ILS Cat I
   - std_alpha_h : 1.0° → 0.35° (~half-dot localizer, ±15m a 3km)
   - std_alpha_v : 0.4° → 0.25° (~half-dot glide slope)

3. Std attitude cales approche stabilisee A320
   - std_yaw   : 2.5° → 1.2°
   - std_pitch : 2.0° → 0.7°
   - std_roll  : 5.0° → 2.5°

4. Couplage roll-lateral amorti (nouveau)
   - roll += gain * derivee_filtree(alpha_h)
   - Filtre EMA passe-bas (alpha=0.15, ~1Hz a 10fps)
   - Gain 2.0, applique apres rate-limit alpha_h, avant rate-limit roll
   - v3 avait echoue avec derivee brute → oscillations excessives

Resultats (turb_high_10fps, v1 → v4) :
  Dev laterale max : 83m → 34m
  Yaw std          : 2.09° → 1.11°
  Pitch std        : 1.58° → 0.60°
  Roll max         : ±10.4° → ±5.0°

Phase 10 — Integration X-Plane 12 (remplacement GES)
================================================================
Date : 2026-03-17

Objectif :
- Remplacer Google Earth Studio (etape manuelle) par X-Plane 12
  pour un rendu 100% automatise (zero intervention humaine)
- Pipeline full auto : XML → TAF → trajectoire → X-Plane rendu → fautes → YOLO → IoU

Problemes resolus :
1. Positionnement camera
   - DREF lat/lon (float32) pas assez precis pour distinguer poses proches
   - DREF elevation read-only, ecrase par ground clamping
   → Solution : pause sim + override planepath + coordonnees locales OpenGL
     (local_x/y/z en metres, precision parfaite)
   - Conversion lat/lon → local via point de reference lu au setup

2. Capture d'ecran
   - Screenshot X-Plane : ~300ms/frame + PermissionError fichiers verrouilles
   → Solution : mss (screen grab Python) cible fenetre X-Plane
     ~30ms/frame en JPEG, 10x plus rapide
   - DPI-aware (SetProcessDPIAware) + GetClientRect pour zone client pure

3. Resolution
   - X-Plane impose minimum 1280px de large (pas de 1024x1024 carre)
   → Resolution finale : 1280x1024 (X-Plane), 1024x1024 (GES)

4. Nommage images
   - LARD attend {name}_{i:0Nd}.ext (0-based, padding dynamique)
   - Corrige dans render_scenario() pour compat LARD export_labels

5. Reticule curseur
   - Mouse yoke actif par defaut (pas de joystick connecte)
   → override_joystick + mouse_is_yoke=0 + deplacement souris hors fenetre

Architecture :
- Dual renderer : --renderer ges (defaut, retrocompatible) | xplane (auto)
- poses.json : format universel (GES + X-Plane)
- run_config.json : stocke le renderer par run
- LARD_RENDERER env var : passe le renderer a Export.py (appele par TAF)

Phase 11 — Effets meteo X-Plane 12 pilotes par XML
================================================================
Date : 2026-03-26

Objectif :
- Permettre a l'utilisateur de definir des conditions meteo X-Plane
  directement dans le XML TAF, avec le meme pattern que les fautes capteur
  (severity + from_pct + to_pct)
- Les effets sont appliques frame par frame pendant le rendu X-Plane
  via datarefs UDP (pas en post-traitement comme les fautes capteur)

Diagnostic datarefs (test_weather_drefs.py) :
  Beaucoup de datarefs meteo sont read-only dans XP12 (nouveau moteur meteo).
  Testes : snow_percent, visibility_reported_m, runway_wetness, thunderstorm_percent
  → tous FAIL (non-ecrivables).
  Ecrivables confirmes : rain_percent, cloud_base/tops_msl_m, temperature_sealevel_c,
  wind_speed_kt, turbulence (mais turbulence inutile car sim en pause).

3 effets meteo implementes (confirmes ecrivables XP12) :
  - rain        : pluie (0-100%), gouttelettes + atmosphere grise
  - cloud_low   : nuages bas, base 500m→50m AGL (simule brouillard/couvert)
  - temperature : temperature 15°C→-15°C (froid, neige possible si combine avec rain)

Mapping severity [0,1] → valeur dataref X-Plane :
  - rain        : severity * 100 → rain_percent (0-100)
  - cloud_low   : base = 500 - severity*450 → 500m (haut) a 50m (ras du sol)
                   top = base + 500m (epaisseur fixe)
  - temperature : 15 - severity*30 → 15°C (doux) a -15°C (froid)

Architecture (meme pattern que les fautes capteur) :
- XML : {type}_severity, {type}_from_pct, {type}_to_pct + contraintes z3
- TAF/z3 sample dans les plages, valide from < to
- Export.py lit les 4 types depuis KNOWN_WEATHER_TYPES, filtre severity > 0
- weather_profile.json sauvegarde pendant generate, charge pendant rendu
- xplane_bridge.py applique les effets frame par frame via send_dref()
- Reset meteo (clair) a la fin du rendu

Difference cle avec les fautes capteur :
  - Fautes capteur : post-rendu, degradation image (OpenCV, footage/ → degraded/)
  - Meteo X-Plane  : pendant le rendu, modifie la scene 3D (datarefs UDP)
  - Les deux se cumulent et n'affectent pas la GT (geometrie piste inchangee)
